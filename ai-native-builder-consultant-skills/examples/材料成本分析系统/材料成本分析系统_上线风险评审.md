## 三、六大模块检查结果

### Section 1 — 模型行为

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 1.1 Vibe check | ⚠️ 待执行 | 文心4.5-28B 本地部署后，必须用20个真实月报场景做 vibe check |
| 1.2 Golden dataset 回归测试 | ⚠️ 待建立 | 需先完成历史数据入库，构建评估集（见 Data Foundation Plan），再建立回归测试 |
| 1.3 对抗性输入测试 | ⚠️ 待执行 | 需测试：财务经理问超出范围的问题时 Agent 是否正确拒绝；用户尝试绕过 Zone C 限制时 Agent 是否坚守边界 |
| 1.4 幻觉风险 | ✅ 已设计缓解 | 趋势预测强制走 RAG + 推理 SOP，不依赖模型参数记忆；置信度低时主动标注，不输出假自信结论 |

**关键风险：** 文心4.5-28B Thinking 模式在财务数据推理场景的实际表现未经验证。Demo 前必须用真实历史数据跑通至少 3 个完整月报周期。

---

### Section 2 — 信任区与 HITL

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 2.1 Zone A 可逆性 | ✅ 已设计 | Zone A 全部为只读操作（SAP API 读取、数据采集、清洗），无写入或发布动作，天然可逆 |
| 2.1 Zone A 审计日志 | ⚠️ 待实现 | 系统设计中已规划，需在代码层落实 |
| 2.2 Zone B 确认 UI | ⚠️ 待测试 | 审核 Web 界面需财务经理真实操作测试；超时行为（财务经理当天未响应）需明确定义 |
| 2.2 幂等性保护 | ⚠️ 待实现 | 财务经理双击确认不得触发重复计算，需在代码层实现 |
| 2.3 Zone C 升级路径 | ✅ 已设计 | Zone C 事件（发布、供应商建议、对外披露）均要求财务经理主动操作，Agent 不介入 |

**关键风险：** Zone B 的邮件通知 + Web 审核的交互链路（点击邮件链接 → 打开审核页面 → 逐节点确认 → 返回 Agent 继续）是最容易在实际使用中出现体验问题的环节，必须在 Demo 前完整跑通。

---

### Section 3 — 失败模式

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 3.1 LLM 不可用时的降级 | ⚠️ 待设计 | 内网 vLLM 服务不可用时，需有明确的失败提示和人工介入通知机制 |
| 3.1 RAG 检索失败时的降级 | ⚠️ 待设计 | pgvector 检索失败时，Agent 需降级到无 RAG 模式并主动告知财务经理"本次未参考历史月报" |
| 3.1 工具调用失败 | ⚠️ 待设计 | SAP API 调用失败、PaddleOCR 识别失败时的处理逻辑需明确，不能静默失败 |
| 3.3 响应超时 | ⚠️ 待设计 | 每个推理步骤需设置超时阈值，避免 Agent 无限等待 |

**关键风险：** 这个系统是月度批处理，失败的代价比实时系统低——但失败必须被立刻发现，不能到财务经理发现月报没生成才知道出了问题。监控和告警必须在 Demo 前就位。

**回滚方案设计（必须在 Demo 前测试）：**

```
触发条件：
  · 月报生成错误率 > 20%
  · 财务经理判断 Agent 输出不可信
  · 任何 Zone C 边界被穿透

回滚动作：
  1. 技术负责人关闭 Agent 自动触发任务（预计 5 分钟）
  2. 通知财务经理"本月切换人工模式，原因：[说明]"
  3. 财务经理按原有人工流程完成月报
  4. 技术团队排查问题，修复后重新评审再恢复

恢复时间目标：< 30 分钟（从决定回滚到人工模式就绪）

通知对象：财务经理、直属领导、技术负责人
```

## 五、最终判断

**🟡 Conditional Go（Demo 阶段）**

5 个 Blocker 全部在 5 月 18 日前解决，Demo 可以按时进行。

**Demo ≠ 上线。** Demo 之后需要：
1. 进入影子模式，积累 ≥3 个月真实月报对比数据
2. 财务经理确认 Agent 输出质量达到"可以替代人工"的标准
3. 完成正式 Go/No-Go 评审

**最高风险项排序：**
1. 🔴 GPU 硬件（Week 1 前未确认 = 整个项目延期）
2. 🔴 阈值标定（Week 2 前未完成 = Zone B 无法跑通）
3. 🟡 5秒响应（需实测，估算不算数）
4. 🟡 Zone B UX（财务经理演练可能暴露交互设计问题）
