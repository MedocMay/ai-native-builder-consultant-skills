# Tech Stack Decision
## 基于 AI 的多维度动态材料成本分析系统

**产品：** 材料成本月报 Agent
**阶段：** MVP → Demo（目标：2026年5月底）
**日期：** 2026-04-13
**版本：** v1.0

---

## 一、硬约束（先排除，再选择）

| 约束 | 值 | 排除的选项 |
|------|-----|----------|
| 数据合规 | 数据不出内网 | 所有外部云端 API（OpenAI、Claude、通义千问云端版） |
| 部署环境 | 公司内网服务器 | 云原生托管服务（AWS Bedrock、阿里云百炼） |
| 对话响应时间（p95） | < 5秒 | 纯串行多轮推理链（需并行化设计） |
| 主模型 | 文心4.5-28B-VL-Thinking（已定） | 其他基础模型选型（锁定） |
| OCR | PaddleOCR-VL（已有） | 其他 OCR 方案（复用现有，不重建） |
| 开发阶段 | MVP Demo，5月底交付 | 过度工程化的生产级框架（付不起的复杂度代价） |

## 二、模型选型

| 用途 | 推荐模型 | 理由 | 何时使用 |
|------|---------|------|---------|
| 主推理模型 | 文心4.5-28B-VL-Thinking（本地） | 已定，VL 支持图表理解，Thinking 支持推理链显性化——正好对应 Zone B 推理 SOP | 趋势预测、差异原因分析、推理链生成 |
| 结构化提取 | 文心4.5-28B（同模型，关闭 Thinking） | 结构化输出任务不需要 Thinking 模式，关闭后速度更快、成本更低 | SAP 数据字段对齐、Excel 入库解析、JSON 输出 |
| OCR | PaddleOCR-VL（本地，复用） | 业内 SOTA，已跑通，不重建 | 中华商务网价格识别 |
| 对话（一问一答） | 文心4.5-28B（流式输出） | 流式输出可以让用户感知到 <2s 首 token，即使完整响应需要5s | 财务经理人机对话 |

**⚠️ 5秒响应的关键处理：**
28B 模型在内网服务器上完整推理可能超过5秒。应对策略：
1. **流式输出（Streaming）**：对话场景必须开启，首 token < 2s 用户体验可接受
2. **Thinking 模式按需开启**：只在需要推理链的 Zone B 场景开启，对话场景关闭
3. **批处理与实时分离**：月报生成走批处理（无延迟要求），对话走实时流式（有延迟要求）
4. **GPU 显存评估**：28B 模型需要至少 2×80GB A100 或 4×40GB A100，部署前必须确认服务器硬件

## 三、框架选型

| 层 | 选择 | 不选其他的原因 |
|----|------|-------------|
| Agent 编排 | **轻量自定义（基于官方 SDK）** | MVP 阶段工作流固定（月度触发→数据采集→分析→审核→月报），不需要 LangGraph/OpenClaw 的动态编排能力；AI coding 环境下自定义代码可维护性更高 |
| 数据管道 | **Apache Airflow（轻量）或简单定时任务** | 月度触发频率低，不需要复杂的流式处理；Airflow 提供可视化监控，方便 demo 演示 |
| RAG / 知识检索 | **pgvector（PostgreSQL 扩展）** | 历史月报、SOP 文档数量有限（<1000份），不需要 Milvus 的分布式能力；PostgreSQL 同时承担业务数据存储，减少运维复杂度 |
| 结构化数据存储 | **PostgreSQL** | SAP 导出数据、Excel 入库数据、标注记录，全部结构化，关系型数据库足够；内网部署，运维简单 |
| 月报生成 | **Python-pptx** | 直接操作 PPTX 模板，财务经理的19页月报格式可以固化为模板，AI 只填充内容；比让 LLM 从头生成 PPTX 更可控、更稳定 |
| 审核界面（Zone B） | **轻量 Web 应用（FastAPI + 简单前端）** | 财务经理需要在浏览器里查看推理链、点击确认；邮件只做通知，实际审核在 Web 界面完成 |
| 邮件通知 | **SMTP（内网邮件服务器）** | 数据不出内网，使用公司内部邮件服务器 |
| 异常检测规则引擎 | **自定义规则层（Python）** | 阈值规则简单明确，不需要 ML 模型；规则用代码写清楚，可审计、可修改 |

## 四、系统架构（MVP 阶段）

```
┌─────────────────────────────────────────────────────┐
│                    触发层                            │
│         月度定时任务（每月中旬固定日期）               │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│                    数据采集层                         │
│  SAP API ──→ 物料成本/BOM/EWO                        │
│  PaddleOCR-VL ──→ 中华商务网价格识别                  │
│  PostgreSQL ──→ 历史 Excel 入库数据                   │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│                  异常检测层（Zone B 入口）             │
│  规则引擎：价格波动阈值 / SAP-Excel 差异 / EWO 变更   │
│  ──→ 异常项：邮件通知 + Web 审核界面等待确认           │
│  ──→ 正常项：进入分析层                               │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│                    分析推理层                         │
│  文心4.5-28B-VL-Thinking（本地）                     │
│  · 成本差异分析（结构化输出，关闭 Thinking）           │
│  · 趋势预测推理链（开启 Thinking，输出 SOP）           │
│  · 差异原因标注（桥型图文字生成）                      │
│  pgvector ──→ 历史月报 RAG 检索（参考历史表达）        │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│               Zone B 审核层（财务经理）               │
│  Web 界面：推理 SOP 逐节点确认                        │
│  邮件通知 ──→ 财务经理点击链接进入审核                 │
│  确认记录 ──→ 自动写入标注数据库（PostgreSQL）         │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│                   月报生成层                          │
│  python-pptx ──→ 填充19页月报模板                    │
│  生成完毕 ──→ 邮件通知财务经理最终审阅                 │
│  发布权：财务经理手动操作（Zone C）                    │
└─────────────────────────────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│                 一问一答对话层（独立）                  │
│  文心4.5-28B 流式输出（关闭 Thinking）                │
│  首 token < 2s，完整响应 < 5s                        │
│  pgvector ──→ 历史数据 RAG 检索                      │
│  数据查询 ──→ PostgreSQL 直查                        │
└─────────────────────────────────────────────────────┘
```

## 五、基础设施

| 组件 | 选择 | 升级触发条件 |
|------|------|------------|
| 推理服务器 | 内网 GPU 服务器（≥2×80GB A100 或同等） | 响应时间 p95 > 5s 时考虑量化或升级硬件 |
| 模型服务 | vLLM（本地推理服务） | 并发用户 > 10 时评估多实例 |
| 数据库 | PostgreSQL + pgvector | 历史数据 > 100 万条时评估分库 |
| 文件存储 | 内网文件服务器（月报 PPTX 存档） | 无需升级，月度频率极低 |
| 监控 | 简单日志（结构化 JSON） | Demo 后进入生产时接入 Grafana |

## 七、决策理由（一段话）

这个选型的核心逻辑是：**用已有的（文心模型、PaddleOCR-VL）、用简单的（PostgreSQL 而不是向量数据库集群、定时任务而不是流式管道）、用可控的（python-pptx 模板而不是 LLM 自由生成格式）**。28B 模型的 Thinking 能力是这个方案的核心差异化——它让推理链天然可解释，直接支撑 Zone B 的"显性推理 SOP + 人工逐节点确认"设计，不需要额外的解释性 AI 层。唯一需要认真对待的风险是 5 秒响应约束，通过流式输出 + 批处理/实时分离解决，但前提是 GPU 硬件必须在 Week 1 前确认到位。

---

## 八、重新评估条件

- [ ] GPU 显存不足（< 2×80GB）→ 评估 INT8 量化或更换硬件，这是最高优先级风险
- [ ] 对话 p95 响应时间 > 5s → 评估 Thinking 模式完全关闭 + 专用对话模型
- [ ] 月报生成准确率 < 90%（财务经理修改率高） → 进入数据飞轮，收集标注数据优化
- [ ] Demo 后用户增加 → 评估 vLLM 多实例部署
