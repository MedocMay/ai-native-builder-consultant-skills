# PRD Lite — 最小产品文档

## 咨询链位置

**在 SOP 中：** Module 1.5，在项目定义（`ai-project-definition`）之后，技术路径选型之前。

**核心价值：** PRD Lite 是从"问题定义"到"技术设计"的桥梁。它比项目定义更结构化（有任务流、有输入输出），比完整 PRD 更轻量（聚焦 MVP，不写未来）。

**常见联动：**
- Agent 路径 → `agent-boundary-design`（PRD Lite 是输入）
- DL 路径 → `dl-data-strategy`（PRD Lite 是输入）
- 需要技术选型 → `tech-stack-selection`

---

## PRD Lite 结构（11个模块）

### 模块 1：项目背景

回答"为什么要做这件事"——不是愿景宣言，是现实问题。

**写法：**
- 现在的业务痛点是什么（一句话）
- 为什么现在是合适的时机（条件具备了什么）
- 不做会有什么后果（可选）

---

### 模块 2：目标用户

从项目定义卡直接复制，确保一致。

**格式：** [角色] · [使用频率] · [使用场景/渠道]

---

### 模块 3：典型使用场景

写 1-2 个具体的使用故事，让读者能"看见"用户在使用这个产品：

**格式示例：**
> 财务经理曹玲莉每月 15 日需要生成材料成本月报。现在她要花 3 天收集 SAP 数据、市场价格和各部门 Excel。
> 有了这个系统，她只需要确认 AI 生成的月报草稿，总耗时从 3 天 → 半天。

---

### 模块 4：当前问题与痛点

从项目定义卡直接提取，确保具体、有数据支撑。

---

### 模块 5：核心任务流（4-6 步）

用户使用这个产品的完整流程，每步说明：
- 谁来做（用户 / AI / 系统自动）
- 做什么
- 结果是什么

**格式示例：**
```
Step 1：[系统自动] 每月15日触发，从SAP API拉取数据
Step 2：[系统自动] PaddleOCR抓取市场价格，规则检测异常
Step 3：[用户确认] 财务经理收到邮件，审核异常数据
Step 4：[AI生成]   生成月报草稿（含推理链）
Step 5：[用户审核] 财务经理逐节点确认推理过程
Step 6：[用户操作] 最终发布月报给管理层
```

---

### 模块 6：输入与输出

明确系统的边界：

| 维度 | 内容 |
|------|------|
| **输入** | [数据来源、格式、频率] |
| **输出** | [产出物、格式、接收者] |
| **触发方式** | [用户主动触发 / 定时自动触发 / 事件触发] |

---

### 模块 7：MVP 范围

**第一版只做什么：**
- 明确列出包含的功能（不超过 3-5 个核心功能）

**第一版不做什么：**
- 明确列出排除的功能（每个都要有理由：留给第二版 / 依赖条件未具备 / 不在 MVP 验证范围内）

---

### 模块 8：技术路径初判

从以下选项中勾选推荐路径（可多选）：

| 路径 | 推荐 | 说明 |
|------|------|------|
| 知识库问答助手（RAG） | □ | |
| Workflow 流程助手 | □ | |
| AgentBuilder 低代码原型 | □ | |
| AI IDE 页面原型 / MVP 雏形 | □ | |
| DL 小模型（CV/OCR/时序） | □ | |
| 大小模型连用架构 | □ | |
| 规则 / 检索 / BI 优先 | □ | |
| 当前先不做 | □ | |

**当前最推荐路径：** [一句话说明]

**为什么不是别的路径：** [一句话排除其他选项]

**未来可能升级到：** [可选，说明演化方向]

---

### 模块 9：风险与边界

**技术风险：**
- [风险 1]：[缓解措施]
- [风险 2]：[缓解措施]

**数据风险：**
- [数据质量/可用性风险]

**组织风险：**
- [需要哪些部门配合？有没有阻力？]

**边界说明：**
- AI 做到哪里，人工从哪里接手（Zone 边界）
- 哪些数据不会被 AI 处理（合规/隐私边界）

---

### 模块 10：成功验证标准

从项目定义卡提取，确保一致：

| 指标 | 基线（现在） | 目标（MVP 后） | 测量方式 |
|------|------------|--------------|---------|
| [指标1] | [现值] | [目标值] | [怎么测] |
| [指标2] | [现值] | [目标值] | [怎么测] |

---

### 模块 11：下一步建议

明确下一步的具体行动，有负责人和时间节点：

- [ ] [行动1] → 负责人：[X] · 截止：[日期]
- [ ] [行动2] → 负责人：[X] · 截止：[日期]

---

## PRD Lite 质量红线

在提交前，确认以下内容都已完成：

- [ ] 项目背景用事实说话，没有"赋能""智能化"等空洞词汇
- [ ] 目标用户具体到角色，不是"员工"或"用户"
- [ ] 典型使用场景能被一个没有背景的人读懂
- [ ] 核心任务流每步明确标注谁来做（用户/AI/系统）
- [ ] 输入和输出都有明确的格式和来源
- [ ] MVP 范围的"不做什么"至少有 3 条
- [ ] 技术路径有明确推荐，并说明了为什么不是其他路径
- [ ] 风险和边界至少各有 2 条
- [ ] 成功标准有基线值和目标值，可以客观测量
