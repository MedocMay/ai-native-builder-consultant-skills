# DL Problem Framing — 场景分析与架构路径决策

## 咨询链位置

**在新版 SOP 中：** 所有 AI 落地项目的入口决策模块。无论 Agent 范式还是 DL 范式，都要先经过这里做路径判断。

**常见联动：**
- 路径A（只用DL）→ `dl-data-strategy` → `dl-model-selection` → 具体pipeline
- 路径B（只用大模型）→ `ai-native-vs-ai-enhanced` → `agent-boundary-design`
- 路径C（大小模型连用）→ 先做DL感知层，再设计Agent理解层

---

## 核心决策：三条技术路径

```
业务场景分析
      ↓
┌─────────────────────────────────────────────────┐
│  路径 A：只用 DL 小模型                          │
│  场景：感知任务，高频，实时，标准化               │
│  示例：工业缺陷检测、票据OCR、设备异常检测        │
├─────────────────────────────────────────────────┤
│  路径 B：只用大模型（LLM/VLM）                   │
│  场景：推理任务，低频，需要语言理解和生成         │
│  示例：月报生成、合同审核、知识问答               │
├─────────────────────────────────────────────────┤
│  路径 C：大小模型连用                            │
│  场景：感知 + 理解都需要，两者缺一不可            │
│  示例：质检+原因分析、票据识别+财务入账           │
└─────────────────────────────────────────────────┘
```

## 路径判断四步法

### Step 1 — 识别核心输入类型

| 输入 | 典型形态 | 优先考虑 |
|------|---------|---------|
| 图像/视频 | 摄像头、拍照、扫描件 | DL（CV/OCR） |
| 结构化数值序列 | 传感器、时序数据、历史记录 | DL（时序） |
| 音频 | 语音、录音、声纹 | DL（Speech） |
| 自然语言文本 | 对话、文档、指令 | 大模型 |
| 结构化数据 | 数据库、表格、报表 | 规则/大模型 |
| 以上组合 | 图像+说明、票据+字段 | 大小模型连用 |

### Step 2 — 判断任务性质

**感知任务（DL 小模型擅长）：**
- 有没有？（缺陷存在吗？文字是什么？）
- 是什么？（属于哪个类别？）
- 在哪里？（位置/坐标）
- 多少？（数量统计）
- 正常吗？（异常检测）

**理解/推理任务（大模型擅长）：**
- 为什么？（原因分析）
- 怎么办？（建议生成）
- 意味着什么？（语义理解）
- 下一步是什么？（决策支持）
- 写一份报告/回复/方案？（内容生成）

### Step 3 — 频率与实时性判断

| 场景特征 | 推荐路径 |
|---------|---------|
| 高频（每秒/分钟大量处理）+ 实时要求 | DL 小模型（毫秒级推理） |
| 低频（每天/每周）+ 可接受延迟 | 大模型（秒级推理） |
| 高频感知 + 低频分析 | 大小模型连用 |

### Step 4 — ROI 快速判断

**值得投入 DL 的条件（满足至少 2 个）：**
- 任务量大（每天 > 1000 个样本，或 24 小时不间断）
- 人工成本高（每个样本需要专家 > 1 分钟）
- 错误代价高（漏检/误判导致质量问题或安全事故）
- 一致性要求高（人工判断受疲劳、经验影响）

**只用大模型的条件：**
- 任务本质是语言理解或生成
- 输入已经是结构化文本/数据
- 频率低，延迟可接受（5-30 秒）

**不需要 AI 的信号（直接 Vibe Coding）：**
- 纯规则统计报表
- 固定格式数据转换
- 能用 SQL + Python 三天内写完的

## 框架选型：PaddleX 之外的选择

**PaddleX 是优秀的全链条平台，但不覆盖所有场景。框架跟着任务走，不是跟着平台走。**

### 完整框架选型地图

| 任务类型 | PaddleX 覆盖 | 主流替代框架 | 推荐模型 |
|---------|------------|------------|---------|
| **CV - 图像分类** | ✅ | PyTorch/Ultralytics | EfficientNet/ConvNeXt |
| **CV - 目标检测** | ✅ | Ultralytics | YOLOv8/YOLOv11/RT-DETR |
| **CV - 语义分割** | ✅ | PyTorch | Segformer/SAM |
| **CV - 实例分割** | ✅ | PyTorch | Mask R-CNN/YOLACT |
| **CV - 工业异常检测** | ✅ STFPM | PyTorch | PatchCore/FastFlow/EfficientAD |
| **CV - 小目标检测** | ✅ | PyTorch | YOLO-NAS/Gold-YOLO |
| **CV - 3D 点云** | ✅ BEV系列 | PyTorch | PointNet++/VoxelNet/CenterPoint |
| **OCR - 通用文字** | ✅ PP-OCRv4 | — | PP-OCRv4 是目前中文最强 |
| **OCR - 表格识别** | ✅ SLANet | PyTorch | TableFormer |
| **OCR - 文档理解** | ✅ PP-ChatOCRv3/v4 | HuggingFace | LayoutLMv3/DocVQA |
| **时序 - 预测** | ✅ DLinear/PatchTST | PyTorch | Informer/Autoformer/iTransformer |
| **时序 - 异常检测** | ✅ PatchTSAD | PyTorch | LSTM-AE/Transformer-AE/TadGAN |
| **时序 - 分类** | ✅ TimesNet | PyTorch | ROCKET/InceptionTime |
| **语音 - 中文识别** | ❌ | PaddleSpeech/FunASR | Paraformer/Conformer |
| **语音 - 多语言** | ❌ | OpenAI | Whisper large-v3 |
| **语音 - 声纹识别** | ❌ | SpeechBrain | ECAPA-TDNN |
| **NLP - 文本分类** | ❌ | HuggingFace | BERT/RoBERTa/ERNIE |
| **NLP - 命名实体识别** | ❌ | HuggingFace | BERT-NER |
| **医疗影像** | ❌ | MONAI | nnUNet/TransUNet |
| **视频 - 动作识别** | ✅ 部分 | PyTorch | TimeSformer/Video Swin |
| **多模态** | 部分 | HuggingFace | CLIP/BLIP2/InternVL |

### 国产硬件适配优先级

```
国产硬件部署（昆仑芯XPU/华为昇腾NPU）
  → 优先用 PaddleX（原生支持）
  → PyTorch 模型需要额外适配工作

NVIDIA GPU（A100/RTX系列）
  → PaddleX 和 PyTorch 都支持
  → PyTorch 生态更丰富，社区更活跃

纯 CPU 推理
  → ONNX 导出 + OpenVINO/ONNXRuntime
  → 两者都支持，看模型导出便利性
```

### 框架选择原则

1. **任务优先**：先找最适合这个任务的模型，再看框架
2. **硬件约束**：国产硬件优先 PaddleX；NVIDIA GPU 两者皆可
3. **团队熟悉度**：团队用什么框架就用什么，不为统一引入陌生栈
4. **商业授权**：确认模型的商业使用授权（Whisper 是 MIT，部分模型有限制）
5. **社区活跃度**：优先选择有持续维护的模型和框架

## 路径 C 大小模型连用标准模板

```
[原始输入（图像/音频/时序）]
        ↓
[DL 小模型感知层]
  · 高速推理（毫秒级）
  · 输出：结构化 JSON
  · 成本：极低（边际成本近零）
  · 框架：PaddleX / PyTorch / HuggingFace
        ↓
[结构化中间结果]
  例：{type: "crack", confidence: 0.94, location: [...]}
        ↓
[触发条件判断]
  · 仅在发现异常/需要解释时调用大模型
  · 控制大模型调用频率 = 控制成本
        ↓
[大模型理解层]（按需调用）
  · 推理分析（秒级）
  · 输出：自然语言报告/建议
  · 模型：文心/Qwen/本地部署 LLM
        ↓
[最终业务输出]
```

**连用场景速查：**

| 业务场景 | 小模型负责 | 大模型负责 |
|---------|---------|---------|
| 工业质检+报告 | CV检测缺陷位置类型 | 分析原因，给处置建议 |
| 票据识别+入账 | OCR提取字段金额 | 理解业务含义，触发流程 |
| 设备监控+运维 | 时序异常检测 | 诊断原因，推荐维修方案 |
| 材料成本预测+分析 | 时序预测未来成本 | 分析趋势原因，给优化建议 |
| 语音记录+摘要 | 语音识别转文字 | 提取关键信息，生成摘要 |

## 常见反模式

**反模式 1："用大模型直接分析图片替代小模型"**
VLM 能理解图片，但速度慢（秒级 vs 毫秒级）、成本高、工业专业场景精度不如专门训练的小模型。正确做法：小模型感知，VLM 理解。

**反模式 2："PaddleX 没有就不能用 DL"**
框架跟着任务走。语音识别用 FunASR/Whisper，NLP 分类用 HuggingFace，医疗影像用 MONAI——选最合适的工具。

**反模式 3："大小模型连用，每次感知都调大模型"**
大模型有成本。小模型高频运行，大模型只在"需要解释/分析"时触发。每张图都调大模型 = 成本失控。

**反模式 4："这个场景一定要用 AI"**
纯统计报表、规则明确的数据处理、固定格式文档生成 → 不需要 AI，Vibe Coding 直接实现更快更稳。
